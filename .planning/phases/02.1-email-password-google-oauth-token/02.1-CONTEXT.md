## Phase Boundary

新增外部使用者登入後端：Email/Password、Google OAuth、使用者與憑證儲存、token 交換與刷新（Phase 02.1 插入階段）。不包含外部 Web UI 與通知/下載體驗前端（Phase 3），也不包含更進階的使用者管理功能。

<decisions>
## Implementation Decisions

### 帳號/身分模型
- **D-01:** 使用者主鍵為系統產生 `user_id`（UUID），email 作為登入識別。
- **D-02:** Google OAuth 與 Email/Password 以 email 自動綁定同一帳號。
- **D-03:** 帳號狀態需支援 `active/disabled`。
- **D-04:** 外部使用者預設 `role=external`。

### Email/Password 註冊/登入
- **D-05:** 不開放自助註冊；帳號由管理者手動建立。
- **D-06:** 不做 Email 驗證；改由管理者手動啟用帳號。
- **D-07:** 忘記密碼/重設密碼延後（本 phase 不做）。
- **D-08:** 密碼規則：至少 12 碼 + 大小寫 + 數字。

### Google OAuth 登入
- **D-09:** Google OAuth 登入時同步基本資料（姓名、頭像、email）。
- **D-10:** 若 email 與既有帳號相同，自動綁定。
- **D-11:** 不限制 Google 帳號網域。
- **D-12:** 不支援取消綁定 Google（延後）。

### Token 與 Refresh 行為
- **D-13:** Access token 有效期 15 分鐘。
- **D-14:** Refresh token 有效期 30 天。
- **D-15:** Refresh 交換時必須輪替 refresh token（舊 token 立即失效）。
- **D-16:** 登出/撤銷時，撤銷該使用者所有 refresh token。

### Claude's Discretion
- 密碼雜湊演算法與參數選擇。
- Login/refresh API 的錯誤訊息與回應格式細節（在不違反既有 API 風格的前提下）。
- 允許同一使用者多裝置同時登入與 refresh token 數量上限。

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `api/auth.py`：JWT 產生/驗證、refresh token 期限計算與 hash。
- `api/models.py`：既有 `RefreshToken`、`ApiKey` 模型可延伸。
- `api_server.py`：已存在 `/api/auth/token`、`/api/auth/refresh`、`/api/auth/revoke` 端點。

### Established Patterns
- Bearer token 驗證使用 `HTTPBearer` + `verify_token`（見 `api/routers/tasks.py`）。
- Refresh token 以 hash 儲存並可撤銷（`pipeline/queue/repository.py`）。

### Integration Points
- FastAPI 主入口在 `api_server.py`，新登入相關端點需掛載於此。
- DB 透過 `pipeline/queue/database.get_session` 管理。

</code_context>
